Blueprint d'implémentation

Que faire quand une version casse

Guide pas à pas, niveau débutant, pour publier une mise à jour Proxima Meet sans risque — et pour réagir si une version déjà déployée est mauvaise.

Trois règles d'or

1. On ne rollback pas une version, on avance : toute correction porte un numéro strictement supérieur.  2. Le rollout progressif est la vraie assurance anti-casse.  3. On protège la clé de signature aujourd'hui, pas après l'incident.

Les 2 cas

CasQuoi faire
Bug fonctionnel récupérableL'app et l'updater fonctionnent encore. On corrige, on republie une version corrigée sous un numéro supérieur : tout le parc l'attrape automatiquement.
Updater cassé / clé perdue non récupérable à distancePlus aucun ordre distant ne passe. Aucun correctif ne peut être poussé : intervention manuelle poste par poste. La prévention (page 2) est donc obligatoire, pas optionnelle.

Ce qui est en place aujourd'hui

Endpoint de mise à jour

https://proxima-meet-updates.s3.fr-par.scw.cloud/{target}/latest.json — un seul bucket, aucun fallback.

Dossiers par OS

windows/ et darwin/, clés windows-x86_64 et darwin-aarch64 dans latest.json.

Signature — fonctionne

minisign, keyID 3432F4938CF02F83, passphrase vide. Le 0.5.37 a été signé et validé le 09/07/2026.

Un seul bucket, une seule clé

Aucune rotation possible aujourd'hui (Tauri n'accepte qu'une pubkey). C'est le sujet de la page 4.

Prévenir : publier une version proprement

Chaque étape = une commande à copier-coller. À suivre dans l'ordre, sans en sauter.

1

Build

Compile l'app et produit les bundles dans target/release/bundle.

cd frontend && pnpm exec tauri build

L'erreur finale sur TAURI_SIGNING_PRIVATE_KEY est normale : les bundles sont déjà produits avant qu'elle n'apparaisse.

2

Signer l'artefact OTA

Génère le fichier .sig avec la clé de production.

pnpm exec tauri signer sign -f "$HOME/.proxima-keys/proxima-updater-2026.key" -p "" "target/release/bundle/nsis/Proxima Meet_0.5.38_x64-setup.exe"

Sous Windows PowerShell, chemin de la clé : "$env:USERPROFILE\.proxima-keys\proxima-updater-2026.key". Copier le contenu du .sig généré dans latest.json, champ signature, verbatim (aucun espace ni retour à la ligne ajouté).

3

Staging — le test qui compte recommandé

Pousser sur un 2ᵉ bucket proxima-meet-updates-staging. Installer la version N sur un poste de test, la pointer vers ce bucket staging, déclencher la mise à jour, et vérifier que l'app redémarre bien en N+1 et re-check l'updater.

4

Upload en production

Dépose les artefacts Windows sur le bucket public (répéter pour darwin/). Le latest.json s'uploade en dernier, une fois les binaires déjà en ligne : sinon le manifeste pointe vers un artefact encore absent, et les clients reçoivent une 404 en tentant de le télécharger.

aws s3 cp ota-package/windows/ s3://proxima-meet-updates/windows/ --recursive --exclude "latest.json" --acl public-read --endpoint-url https://s3.fr-par.scw.cloud aws s3 cp ota-package/windows/latest.json s3://proxima-meet-updates/windows/latest.json --acl public-read --endpoint-url https://s3.fr-par.scw.cloud
5

Vérifier que c'est public

Doit renvoyer le bon numéro de version, en HTTP 200.

curl https://proxima-meet-updates.s3.fr-par.scw.cloud/windows/latest.json
6

Rollout progressif

Avec un seul latest.json public, tout le parc se met à jour d'un coup dès qu'il est en ligne. Pour un vrai rollout progressif : (a) pointer d'abord quelques postes pilotes vers le bucket staging (étape 3), valider 24 h ; (b) publier ensuite le latest.json en production pour tout le parc.

À retenir

Valider l'app en local ≠ valider l'OTA. Le seul vrai test, c'est la transition N→N+1 (étape 3) — un bug d'updater ne se voit jamais autrement.

Récupérer : une version déjà chez les clients est mauvaise

Arbre de décision à suivre dans l'ordre. Une seule question détermine tout le reste.

1

L'app se lance encore ?

OUI → bug fonctionnel, récupérable, on continue ci-dessous. NON → cas catastrophe : voir page 4 (protection de la clé et de la fondation).

2

Corriger et bumper la version

Corriger le code, puis monter le numéro de version strictement au-dessus (ex. 0.5.37 → 0.5.38) dans tauri.conf.json et package.json.

3

Rebuild, re-signer, republier

Refaire l'étape 1 puis l'étape 2 de la page 2 (build + signature), puis l'upload et le latest.json. Les clients passent à 0.5.38 automatiquement, sans action de leur côté.

4

Règle absolue

Ne jamais republier un numéro égal ou inférieur. L'updater ne le proposera pas — la mise à jour serait silencieusement ignorée par tout le parc.

Script pré-déploiement — bloquant

# vérifie : version strictement croissante + signature valide, avant toute promotion
NEW=$(jq -r .version new_latest.json)
CUR=$(curl -sf https://proxima-meet-updates.s3.fr-par.scw.cloud/windows/latest.json | jq -r .version)
[ "$(printf '%s\n%s\n' "$CUR" "$NEW" | sort -V | tail -1)" = "$NEW" ] && [ "$NEW" != "$CUR" ] \
  || { echo "REFUS : $NEW n'est pas strictement supérieur à $CUR"; exit 1; }
minisign -Vm "Proxima Meet_0.5.38_x64-setup.exe" -x "Proxima Meet_0.5.38_x64-setup.exe.sig" -P RWSDL/CMk/QyNBehdAlxTbTT6Yos0lab6gjrqBZqZ3X8IoXcazjTRWqc \
  || { echo "REFUS : signature invalide"; exit 1; }
echo "OK — déploiement autorisé"

Script post-déploiement — validation

# le manifeste doit être public et répondre 200
curl -sf -o /dev/null -w '%{http_code}' https://proxima-meet-updates.s3.fr-par.scw.cloud/windows/latest.json
# l'artefact doit exister réellement (HEAD, pas de page d'erreur)
curl -sfI https://proxima-meet-updates.s3.fr-par.scw.cloud/windows/Proxima%20Meet_0.5.38_x64-setup.exe

Protéger la fondation (à faire cette semaine)

Le risque réel

Perdre le fichier-clé, c'est plus jamais de mise à jour sur tout le parc, sans recours possible.

1

Sauvegarder la clé

Copier ~/.proxima-keys/proxima-updater-2026.key (et le .pub) dans Proton Pass (coffre équipe), plus une copie hors-ligne. Noter que la passphrase est vide. keyID : 3432F4938CF02F83.

2

Se rendre capable de rotation (dual-key, modèle Sparkle)

Préparer une 2ᵉ clé pendant que la clé A est encore disponible — pas après l'avoir perdue.

VersionSignée avecEmbarque
N (déjà déployée)Clé APubkey A uniquement
N+1Clé A (les clients l'acceptent encore)Pubkey A + pubkey B
N+2Clé BSeuls les clients ≥ N+1 l'acceptent

Règle : ne jamais changer la clé et l'endpoint dans la même release. Ajouter un 2ᵉ endpoint (fallback) dans tauri.conf.json avant d'en avoir besoin.

Checklist finale

Clé sauvegardée dans Proton Pass
Passphrase (vide) documentée
2ᵉ bucket de staging créé
Recette OTA intégrée à chaque release
Rollout progressif appliqué
Scripts pré/post-déploiement en place
Endpoint de fallback ajouté
2ᵉ clé prête (dual-key)
